## Project Context (One-Paragraph Summary)

Datastream Books is an internal finance/accounting application replacing the legacy Macola system. Built on Microsoft Dataverse (model-driven app) + Azure SQL (immutable ledger). Multi-entity from day one. Sister application to Datastream ERP (operations). Finance System Owner is Pam. Built with audit-defensibility as a core architectural requirement, not an afterthought.

## Operating Principles

### Be Direct and Outcome-Focused

- Provide complete, ready-to-use outputs — not partials
- Default to full-file replacements over diffs
- Make reasonable assumptions and proceed; surface assumptions explicitly
- Optimize for speed, maintainability, and real-world use

### Server-Side Logic Wins

Use **Dataverse plugins (C#)** for business logic. Use Power Automate only for:
- Simple notifications
- Scheduled jobs outside transaction context
- Third-party connectors without C# SDKs

Plugins are preferred because they offer atomicity, performance, testability, version control, and audit defensibility.

### Immutability Is Architectural, Not Promised

Financial controls live in the data layer and plugins, not in honor systems or UI:
- Append-only Azure SQL ledger with `DENY UPDATE, DELETE` at SQL role level
- Hash-chained records (SHA-256)
- Period locks enforced by plugins
- Segregation of Duties enforced by plugins (`CreatedBy != ApprovedBy`)
- Server-side posting only — no direct table writes from UI

See `docs/architecture/immutability-design.md` for full specification.

### Multi-Entity from Day One

Every transactional table includes `EntityId`. Every report includes Entity selector. Inter-company JE pairs auto-generated by plugins. Never assume single-entity.

## Reference Data

The repo holds two kinds of reference material under `docs/reference/`:

1. **Books reference data** — chart-of-accounts seed CSVs, fiscal-calendar
   seeds, and other inputs the build will use to populate Datastream Books.
   These get loaded into Books and become part of the system.

2. **ERP metadata reference** — a *read-only* snapshot of the
   PRI-Datastream (ERP) solution's table, column, and relationship metadata,
   under [`docs/reference/erp-metadata/`](docs/reference/erp-metadata/).
   This describes the ERP solution, **not** Datastream Books. It is here
   because Books shares the `rm` publisher with ERP and we want to follow
   the naming and shape patterns the operations team is already using.

### Rules for `docs/reference/erp-metadata/`

- ⛔ **These files are NOT the Books schema.** Do not implement these tables
  in Books. Do not copy column definitions into `solution/src/`.
- ⛔ Do not run any `pac`, SQL, or Power Platform command that mutates ERP
  based on these files. They are a point-in-time snapshot.
- ✅ Read freely when designing Books tables — match patterns where it
  makes sense (master-data column shape, naming, etc.).
- ✅ Refresh as-needed (not scheduled). Add a dated row to the file inventory
  in `docs/reference/erp-metadata/README.md` on every refresh — don't
  silently overwrite.
- ✅ For the patterns extracted from these files, see
  [`docs/architecture/erp-pattern-notes.md`](docs/architecture/erp-pattern-notes.md).

If a Books table needs to *reference* an ERP table (e.g., AR pointing at
`rm_customer`), that's a Dataverse lookup across solutions — design it as
a read-only relationship and document the ownership boundary in the
table's data-model entry.

## Code Conventions

### C# Plugins

- One plugin class per file
- Plugin name pattern: `<Action><Entity>Plugin` (e.g., `PostJournalEntryPlugin`)
- Implement `IPlugin` interface
- Register via solution (not via Plugin Registration Tool manually)
- Unit tests in parallel folder under `plugins/DatastreamBooks.Plugins.Tests/`
- Use `FakeXrmEasy` (or equivalent) for plugin unit testing
- Wrap all critical logic in try/catch with meaningful logging
- Never swallow exceptions silently

### TypeScript / React (PCF + Custom Pages)

- Use Fluent UI v9 components for consistency with model-driven app look
- Strict TypeScript (no `any` except with explicit comment)
- Functional components with hooks; no class components
- Standard event handlers (`onClick`, `onChange`) — NEVER use HTML `<form>` tags in PCF
- Component file: `ComponentName.tsx`
- Test file: `ComponentName.test.tsx`

### SQL Migrations

- File naming: `V<NNNN>__<description>.sql` (Flyway-style, four-digit number)
- One migration per concern
- Never modify a committed migration — always add a new one
- Every migration must be idempotent OR wrapped in a transaction
- Include rollback comments at the bottom of each migration
- Append-only tables (`GeneralLedgerEntries`, `AuditEvents`, etc.) must include the `DENY UPDATE, DELETE` grants in their creation migration

### PowerShell Scripts

- Use approved PowerShell verbs (`Get-`, `Set-`, `New-`, `Invoke-`, etc.)
- Parameter validation at the top of every function
- Use `Write-Host` for user messages, `Write-Verbose` for debugging, `Write-Error` for errors
- All scripts must support `-WhatIf` where applicable
- Never hardcode environment URLs or credentials — read from auth profiles or env vars
- **ASCII-only source, UTF-8 with BOM on disk.** No em-dashes (use `--`), no section signs (write the word `section`), no smart quotes. Save as UTF-8 *with* BOM so Windows PowerShell 5.1 reads the file deterministically — non-BOM UTF-8 is read using the legacy code page and any non-ASCII byte becomes mojibake that breaks the parser. Both safeguards belt-and-suspenders: ASCII source means even a no-BOM accident is harmless; the BOM means even a non-ASCII slip is parseable. Verify new scripts with `powershell.exe -NoProfile -File <path> -WhatIf` before committing. Codified in [`docs/decisions/datastream-books-decisions.md`](docs/decisions/datastream-books-decisions.md) §64.

## Build and Test Commands

> These commands assume you are in the repo root.

### First-Time Setup

```powershell
# Verify pac CLI
pac --help

# Verify auth profiles
pac auth list

# Restore .NET dependencies
dotnet restore

# Install Node dependencies for PCF/pages
npm install --prefix pcf
npm install --prefix pages
```

### Pull Solution from Dev

```powershell
.\scripts\pull-solution.ps1
```

This exports the current solution from `pri-books-dev`, unpacks it, and updates the `solution/` folder for commit.

### Build Plugins

```powershell
dotnet build plugins/DatastreamBooks.Plugins/DatastreamBooks.Plugins.csproj
```

### Run Plugin Tests

```powershell
dotnet test plugins/DatastreamBooks.Plugins.Tests/DatastreamBooks.Plugins.Tests.csproj
```

### Build PCF Controls

```powershell
cd pcf/<ControlName>
npm run build
```

### Push Solution to Dev

```powershell
.\scripts\push-solution.ps1
```

### Apply SQL Migrations

```powershell
.\scripts\run-sql-migration.ps1 -Environment dev
```

## What NOT to Do

- ❌ **Never make direct changes to `pri-books`** — always flow through Dev and deployment
- ❌ **Never commit secrets** (API keys, connection strings, passwords) — use environment variables / Azure Key Vault
- ❌ **Never commit packed solution zips** — only unpacked source
- ❌ **Never write financial logic in Power Automate** — use plugins
- ❌ **Never use `UPDATE` or `DELETE` on `GeneralLedgerEntries`** — append-only enforced at SQL role
- ❌ **Never assume single-entity** — always include `EntityId` in transactional tables
- ❌ **Never bypass plugin posting** — UI cannot write directly to posted ledger
- ❌ **Never modify a committed SQL migration** — add a new one
- ❌ **Never use HTML `<form>` tags in PCF controls** — use event handlers
- ❌ **Never use `localStorage` / `sessionStorage` in PCF or custom pages** — use Dataverse state
- ❌ **Never skip the Change Request system** for production changes
- ❌ **Never use the Plugin Registration Tool for routine plugin changes** — PRT is for first-time registration of a new assembly or first step on a new entity only. Routine code changes, additional steps on an existing assembly, image edits, and bug-fix redeploys all flow through CI/CD so the change lands in source control with a git commit (and, eventually, a CR). PRT-side changes that aren't followed by `pull-solution.ps1` + commit break the audit-defensibility narrative ("every production-affecting change has a CR and a Git history entry"). See `docs/runbooks/plugin-registration.md` § "When to use" and risk R-A-20.

## Key Principles to Remember

1. **Audit-defensibility is non-negotiable.** Every design decision is evaluated against "could we defend this to an auditor?"
2. **Pam owns the finance system.** IT builds it for her. Her name is on every finance deliverable.
3. **Multi-entity from day one.** No retrofitting later.
4. **Server-side over client-side.** Plugins over Power Automate.
5. **Immutability through architecture.** Not through promises.
6. **Documentation in the repo.** Not in chat, not in email, not in heads.
7. **Verification is mandatory, not optional.** When a prompt or runbook asks you to verify a value — publisher, prefix, environment name, credential, schema — run the verification before proceeding. Do not substitute a value from `AGENTS.md`, training data, or a prior prompt for an actual lookup. Documented values can be stale; live environments are the source of truth. If verification surfaces a discrepancy, **stop and surface it** rather than silently picking one. The cost of an extra `pac` (or equivalent) command is seconds; the cost of an incorrect configuration baked into a solution is hours of cleanup. This principle was added after a real incident on this project where assumed publisher values (`dsb` prefix) had to be reverted to the correct ERP-matching values (`rm` prefix) after a solution rebuild.
